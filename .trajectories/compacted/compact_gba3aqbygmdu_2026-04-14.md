# Trajectory Compaction: 2026-03-24 - 2026-03-28

## Summary
These trajectories span roughly three weeks (March 24–28, 2026) of intensive multi-agent development on the relayfile project — a real-time filesystem for humans and AI agents. The work progressed in three major phases: (1) initial feature buildout including bulk operations, WebSocket events, CLI/DX, CI/CD pipelines, a landing page, and the relayfile-cloud Cloudflare Workers port; (2) a deep refactor extracting business logic from the relayfile-cloud `workspace.ts` monolith into `@relayfile/core` (packages/core), with modules for ACL, events, export, files, operations, query, semantics, storage, tree, webhooks, and writeback; and (3) hardening the Go mount client with FUSE support (`internal/mountfuse/`), fsnotify-based change detection (`internal/mountsync/watcher.go`), permission-enforcing pull filtering, `.relayfile.acl` enforcement in HTTP handlers (`internal/httpapi/acl.go`, `internal/httpapi/auth.go`), and relayauth scope format compatibility. The DAG workflow orchestrator pattern was used consistently: an orchestrator agent reads context files in parallel, converges, fans out implementation to specialist workers, converges again, then runs a fix-build step. Most workflows completed in 1–13 minutes with 2–7 agents. Notable failures occurred in the PR #11 review workflow — two consecutive runs failed at the security-review step with 'Unexpected failure', cascading to skip all downstream cross-review and merge-findings steps. The third attempt succeeded using 7 agents in 12 minutes. Agent coordination happened via Relaycast channels (#general, #wf-fuse-mount), with agents checking in, claiming assignments, and posting step outputs. Key code artifacts: `internal/mountfuse/` (8 files: fs.go, dir.go, file.go, client.go, cache.go, wsinvalidate.go, cache_test.go, fuse_test.go), `internal/mountsync/watcher.go` (fsnotify), `internal/httpapi/acl.go` and `acl_test.go`, `packages/core/src/` (12 modules), `.github/workflows/ci.yml`, `publish-npm.yml`, `release-binaries.yml`, `site/` (Astro + Tailwind landing page with SVG diagrams), and `scripts/e2e.ts` (end-to-end test harness).

## Conventions Established
- **DAG workflow with read-context → converge → parallel-impl → converge → fix-build**: Every successful multi-agent workflow follows this pattern: parallel file reads, orchestrator convergence, fan-out to specialist workers, convergence check, then a single fix-build agent resolves any compilation/test issues (scope: All relay workflow orchestrations)
- **Stacked PRs when base branch hasn't merged to main yet**: Using main as PR base when prior refactor commits aren't merged duplicates diffs and makes review noisy; stacking keeps each round's diff focused (scope: Git workflow for dependent feature branches)
- **Exclude dist/ and generated output from commits; only commit src/ and package metadata**: Repo-level .gitignore excludes dist/, existing packages don't track built artifacts, source is the durable truth (scope: All npm packages in the monorepo)
- **StorageAdapter as the abstraction boundary for @relayfile/core**: Core business logic must stay pure over StorageAdapter so it works with both the Go SQLite store and the CF Workers D1 store; no storage-specific concerns leak into core modules (scope: packages/core/src/*.ts)
- **Build tags for optional FUSE support (fuse_mount.go)**: FUSE depends on platform-specific libraries; build tags keep the base binary compilable without FUSE dependencies (scope: cmd/relayfile-mount/)

## Lessons Learned
- Multi-agent PR review workflows are fragile — all three review agents failed simultaneously twice before succeeding on the third attempt (review-pr-11-workflow failed twice with 'Unexpected failure' at the security-review step, cascading to skip 4 downstream steps each time. The third run with 7 explicitly named agents (security, developer, historian, security-xr, developer-xr, historian-xr, synthesizer) succeeded in 12 minutes) - Add retry logic or fallback paths for review agent steps; investigate the root cause of the 'Unexpected failure' error rather than relying on re-runs
- Agent handoff works when predecessors become unresponsive — verify stub state before implementing (Agent-6 became unresponsive while implementing operations.ts and writeback.ts; the takeover agent first confirmed both files were still stubbed before proceeding, avoiding stomping on partial work) - Always check file state before taking over from an unresponsive agent; treat takeover as a fresh implementation after confirming the baseline
- Fix-build steps consistently catch real issues even when individual impl workers report success (The bulk-and-export workflow's fix-issues step found 4 critical issues including a syncer mutex deadlock (connectWebSocket running inside the lock) and SDK field mismatches, despite all impl workers reporting verification passed) - Never skip the fix-build convergence step; individual worker verification is necessary but not sufficient for integration correctness
- Conformance suite exposed parity gaps that unit tests missed (relayfile-cloud investigation found gaps in wildcard If-Match on write/delete, ACL marker inheritance, websocket catch-up, and absent-row SQL handling — none caught by existing route-forwarding contract tests) - Run the shared conformance suite against both Go and CF Workers implementations in CI; it catches behavioral divergence that provider-specific unit tests miss
- Many standby/acknowledgment sessions produced no work — coordination overhead in multi-agent setups is significant (At least 8 sessions were pure coordination: checking in on Relaycast, acknowledging other agents, confirming standby status. Each was 0-1 minutes but accumulated to meaningful idle time) - Consider reducing coordination ceremony — agents should self-assign from a work queue rather than waiting for explicit assignment via channel messages
