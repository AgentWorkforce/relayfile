## Conventions Established
- **FUSE WebSocket auth token sourcing: always read tokens from the mount HTTP client's current credentials or re-read creds file before reconnect, never use cfg.token directly**: cfg.token is set once at mount startup; credential-file rotation triggers HTTP 401 refresh but does not update cfg.token. WebSocket reconnects can occur before any HTTP operation, so using cfg.token will cause stale-token failures after rotation. (scope: cmd/relayfile-mount/fuse_mount.go, cmd/relayfile-mount/main.go WebSocket invalidator and daemon ticker callback construction)
- **Write-only sync mode: disable WebSocket pull connections in daemon ticker, keep normal reconcile cadence**: Write-only mode is documented to push local changes without mirroring provider history; maintaining WebSocket pull connections violates this contract and wastes resources. (scope: cmd/relayfile-mount/main.go daemon ticker logic for sync-mode=write-only mounts)
- **CLI lagging diagnostics: when status=lagging and lagSeconds=0, query sync ingress state and render reason (no cursor/watermark/ingress vs actual backlog)**: Server can report lagging with zero lag when a provider has no cursor or watermark; without ingress context, users see confusing zero-lag lagging status and cannot distinguish initialization from actual backlog. (scope: cmd/relayfile-cli/main.go relayfile status rendering for providers with status=lagging and lagSeconds=0)
- **Multi-repo SDK workflows: architect writes acceptance contract (API, SDK exports, E2E proof, exit markers) before implementation; sdk-impl, cloud-impl, e2e-impl reference contract for verification gates**: Upfront contract prevents drift between SDK and cloud implementations; explicit E2E proof requirements and exit markers enable automated verification and clear completion signal. (scope: docs/sdk-setup-client-acceptance.md for SDK/cloud contract definition; DAG workflow steps reference contract for pass/fail gates)
- **DAG workflow step verification: each implementation step runs focused tests and verifies exit markers/signals before unblocking dependent steps**: Sequential verification gates catch implementation errors early and prevent cascading failures; completion markers (SDK_SETUP_E2E_OK, SDK_E2E_PROOF_READY, ACCEPTANCE_CONTRACT_READY) provide deterministic pass/fail signal for workflow orchestrator. (scope: All SDK workflow steps (collect-context, define-acceptance-contract, implement-sdk, implement-cloud-contract, run-tests, fix-tests, typecheck, regression, E2E proof))

## Lessons Learned
- WebSocket token callbacks must re-read credentials file before each reconnect, not cache initial config token (PR #252 fixed HTTP client token refresh on 401, but FUSE WebSocket invalidator and daemon ticker callback still used cfg.token from mount startup. WebSocket reconnects can be the first operation after credential rotation, causing stale-token failures that HTTP refresh cannot prevent.) - When adding WebSocket clients to FUSE mount or daemon ticker, always source tokens from a callback that re-reads the creds file or queries the mount HTTP client's current credentials; never capture cfg.token at construction time.
- CLI diagnostics for ambiguous server status require querying additional context (ingress state, cursor/watermark) and rendering inline explanations (relayfile status reported lagging providers with lagSeconds=0, nil cursor, nil watermark, no errors, no dead letters, and no ingress events. Users saw 'lagging' without any visible lag metric and could not distinguish initialization from actual backlog. Adding ingress query and rendering 'no cursor/watermark' or 'no ingress events' disambiguated the confusing case.) - When server status values are ambiguous (lagging with zero lag, healthy with errors), extend CLI to query additional state and render inline diagnostic reasons; do not rely on users to infer meaning from raw status + lag numbers.
- Acceptance-driven multi-repo workflows prevent implementation drift and enable automated verification, but require upfront contract definition before coding begins (SDK workflow used architect agent to write docs/sdk-setup-client-acceptance.md with explicit API contract, SDK exports, E2E proof requirements, and exit markers before sdk-impl or cloud-impl started coding. All 32 DAG steps referenced the contract for pass/fail gates, and the E2E proof verified SDK_SETUP_E2E_OK + SDK_E2E_PROOF_READY markers. First workflow run (16:53-17:05) stopped mid-implementation; second run (17:12-17:23) completed all steps and passed E2E proof.) - For multi-repo feature work (SDK + cloud, SDK + CLI, etc.), always write the acceptance contract first with explicit API surface, export list, E2E proof script, and completion markers. Have architect agent own contract definition, then pass to specialist agents for parallel implementation. Use deterministic exit markers (not subjective 'looks good') for workflow verification gates.
- Write-only sync mode must disable all pull/mirror operations (WebSocket connections, remote history polling) to match documented contract (PR #243 added write-only sync mode to push local changes without mirroring provider history, but daemon ticker was still maintaining WebSocket pull connections. Review caught this violation of the documented sync-mode contract.) - When adding new sync modes or operational modes (read-only, write-only, mirror, etc.), audit all background operations (daemon tickers, WebSocket connections, reconcile loops) to ensure they respect the mode's documented contract; do not assume existing background logic will self-adjust based on mode flag.
