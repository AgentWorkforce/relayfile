# Durable ACKed Writeback Outbox

## Problem

Relayfile command-root writes currently rely on the normal mirror sync path. A
local JSON command file becomes a dirty tracked file, then the mount attempts a
`POST /v1/workspaces/{workspaceId}/fs/bulk` during the same sync cycle. This is
best-effort transport, not a durable outbox:

- the command intent is not assigned a stable mount-side command id;
- pending payloads are represented indirectly as `trackedFile.Dirty`;
- upload completion is inferred from the bulk response and optional follow-up
  `ReadFile`, not from a durable server receipt;
- websocket disconnects only clear the invalidation stream and fall back to
  polling; they do not independently flush command-root sends;
- pull/conflict reconciliation and write-up share the same cycle and lock.

Observed failure mode: the local command file was captured by Pear, but the
mount cycle hit `context canceled` / websocket fallback / conflict storms before
the upload reached the cloud writeback consumer. The file remained local with no
delivery receipt and no surfaced failure.

## Current Drop Points

Go mount client:

- `cmd/relayfile-mount/main.go` creates a per-cycle timeout context in
  `runSinglePollingMount.run` and logs `mount sync cycle failed: %v` without a
  durable retry record beyond normal local state.
- `internal/mountsync/syncer.go` runs `sync`: connect websocket, pull remote,
  `pushLocal`, maybe pull again. A failed `pushLocal` marks sync error and saves
  state, but the send is still just a dirty mirror file.
- `pushLocal` scans all local files, prepares `pendingBulkWrite` values, and
  calls `flushPendingBulkWrites`.
- `flushPendingBulkWriteChunk` calls `RemoteClient.WriteFilesBulk`. If the
  request or follow-up revision lookup is canceled, it returns error before
  `reconcileBulkWrite` can clear `Dirty`.
- If the cloud response omits per-file `results`, `reconcileBulkWrite` performs
  `ReadFile` to discover the revision. This widens the post-commit cancellation
  window: the server may have accepted and queued provider writeback, while the
  mount still cannot record a local ACK.

Cloud receive/dispatch:

- `packages/relayfile/src/routes/fs.ts` forwards `/fs/bulk` to the workspace
  Durable Object after requiring `fs:write` and `X-Correlation-Id`.
- `packages/relayfile/src/durable-objects/handlers/fs.ts` `bulkWrite` stores
  content, writes file rows, records events, creates writeback operations, and
  dispatches to `WRITEBACK_QUEUE`.
- `packages/relayfile/src/durable-objects/handlers/ops.ts` `recordMutations`
  returns per-file `WriteQueuedResponse` values with `opId`, `targetRevision`,
  and writeback state internally, but the public bulk response currently does
  not expose those as mount upload receipts.
- `packages/relayfile/src/queue-consumer.ts` ACKs Cloudflare queue messages
  after `executeProviderWritebackBatch`.
- `packages/relayfile/src/writeback/provider-executor.ts` ACKs provider
  execution to `/writeback/{opId}/ack`; Slack has provider-level idempotency in
  `slack_writeback_idempotency`. This starts after cloud has accepted upload.

## Guarantees

1. Durable outbox: command-root file writes are persisted as outbox entries
   before network I/O, retried until server ACK or bounded retry exhaustion,
   and survive reconnect, restart, and crash.
2. Resumable reconnect: every websocket reconnect and every polling cycle
   flushes pending outbox entries before or independently from mirror pull work.
3. Server ACK receipt: the cloud returns a durable receipt for each accepted
   command with enough data for the mount to mark it delivered and for users to
   inspect stuck sends.
4. Isolation: command outbox flushing is not blocked by sync-down conflict
   reconciliation. Conflict storms may affect mirror convergence, but not
   command dispatch attempts.

## Outbox Model

Persist outbox state under `.relay/outbox/`:

- `.relay/outbox/pending/<commandId>.json`
- `.relay/outbox/acked/<commandId>.json`
- `.relay/outbox/failed/<commandId>.json` for terminal local rejection (such
  as schema validation or permission denial) or a terminal provider receipt.
- `.relay/outbox/attention/<commandId>.marker` when a failed command requires
  operator attention. This auxiliary marker keeps health summaries read-free;
  the complete record remains at the stable `failed/<commandId>.json` path.

`commandId` is a stable id generated by the mount when it first observes a
command-root file. Recommended shape:

```text
mountcmd_<base32(sha256(workspaceId + remotePath + contentHash + firstSeenAt))>
```

Each pending record stores:

- workspace id and remote path;
- content type, encoding, content hash, and body or body file reference;
- first seen time, last attempt time, attempt count, next attempt time;
- stable `correlationId`;
- latest server receipt, if any;
- local file mtime/size used only to detect user edits and supersede the entry.

If the local command file changes before ACK, supersede the old entry with a new
command id and leave the old entry observable as `failed/superseded`.

## Wire Protocol

Add an additive command receipt shape to `/fs/bulk` for agent writes:

```json
{
  "written": 1,
  "errorCount": 0,
  "errors": [],
  "correlationId": "mountcmd_...",
  "results": [
    {
      "path": "/slack/channels/C/messages/...",
      "revision": "rev_123",
      "contentType": "application/json",
      "opId": "op_456",
      "writeback": {
        "provider": "slack",
        "state": "pending"
      },
      "receipt": {
        "commandId": "mountcmd_...",
        "acceptedAt": "2026-06-09T14:00:00Z",
        "dedupeKey": "mountcmd_..."
      }
    }
  ]
}
```

Request additions are also additive:

- top-level `outbox: true` for command-root flushes;
- per-file `commandId`;
- per-file `contentIdentity` using `{ kind: "mount-command", key:
  commandId, ttlSeconds: ... }` so retrying after a lost response is idempotent;
- `X-Correlation-Id` equal to the stable command id or batch id.

Cloud behavior:

- if the same `commandId` is received again after prior acceptance, return the
  original receipt and do not create a duplicate provider operation;
- receipt means the workspace DO durably stored file content, event, and
  writeback operation, and dispatch was either queued or recoverably scheduled;
- provider execution success/failure remains the existing `/writeback/{opId}/ack`
  lifecycle.

## Mount Flow

1. Detect command-root writes separately from ordinary mirror writes.
2. Persist outbox entry before network I/O.
3. Flush pending outbox entries on mount start, websocket connect/reconnect,
   local command write, and periodic tick.
4. Use a root-derived retry context for outbox flush attempts, not the normal
   pull/reconcile cycle context.
5. On receipt, atomically move the outbox record to `acked`, update
   `.relay/state.json`, and clear the dirty tracked file using the receipt
   revision without a follow-up `ReadFile`.
6. On retryable error or context cancellation, keep the pending record and
   schedule bounded backoff. After the retry budget is exhausted, stop retrying
   silently and surface the command as needs-attention in `.relay/state.json`.
7. On terminal cloud rejection, move to `failed`, surface the reason in
   `.relay/state.json`, and preserve the local file.

## Smallest Reliable Increment

Phase 1 should avoid provider executor changes:

1. Add mount-side persistent outbox records for paths under configured command
   roots.
2. Flush those records before `pullRemote` and from websocket reconnect.
3. Treat `/fs/bulk` 202 as server ACK and persist the response correlation id,
   path, revision when present, and any writeback/op fields the server returns.
4. Add cloud bulk response `results[].opId/writeback/receipt` by plumbing the
   existing `recordMutations` `WriteQueuedResponse` values into the public
   response.
5. Add idempotency via `contentIdentity` or equivalent command-id storage so a
   lost response can be retried safely.

Later phases can add CLI surfaces:

- `relayfile writeback outbox list`;
- `relayfile writeback outbox retry <commandId>`;
- `.relay/outbox-status.json` for UI/status consumers.

## Tests

Mount tests:

- local command write persists an outbox file before `WriteFilesBulk` is called;
- context cancellation during `WriteFilesBulk` leaves the entry pending and a
  later flush delivers it;
- cancellation after server commit but before local reconcile is deduped and
  resolves from the server receipt;
- sync-down conflict errors do not prevent outbox flush from being attempted;
- restart reloads pending outbox and flushes before normal polling.

Cloud tests:

- `/fs/bulk` response includes per-file receipt data for agent writes;
- retry with same `commandId` returns the same receipt and does not enqueue a
  duplicate writeback operation;
- provider-level retry/ACK behavior is unchanged.
